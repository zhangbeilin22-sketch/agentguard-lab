# AgentGuard Lab 学习指南

这份文档面向第一次做完整 AI 工程项目的同学。目标不是背代码，而是做到三件事：能运行、能修改、能向面试官解释设计选择。

## 学习前准备

建议具备以下最低基础：

- 看得懂 Python 函数、列表、字典和 `if` 判断；
- 知道 JSON 是一种结构化数据格式；
- 会在 PowerShell 中执行命令；
- 知道 Git 的 `add`、`commit` 和 `push` 分别做什么。

暂时不懂 `dataclass`、类型注解、依赖注入或安全指标没有关系，按照下面顺序边运行边理解即可。

## 第 1 天：先把项目跑起来

执行：

```powershell
cd E:\github\agentguard-lab
.\run.ps1
```

完成三个实验：

1. 基线模式运行“邮件诱导外发密钥”，确认攻击成功。
2. 平衡模式运行同一场景，确认攻击被阻断且摘要仍然创建。
3. 严格模式运行“发送项目进展”，确认合法邮件等待审批。

今天需要能回答：

- 什么是提示词注入？
- 为什么普通文本能够影响 Agent 的工具调用？
- “检测到攻击”和“真正阻止攻击”有什么区别？

## 第 2 天：理解场景数据

打开 `data/scenarios.json`，找到 `attack_email_exfiltration`。

重点理解：

- `user_request`：用户真正想完成的任务；
- `untrusted_content`：邮件、文件或知识库返回的数据；
- `expected_actions`：正常业务动作；
- `injected_actions`：攻击者希望模型执行的动作；
- `allowed_tools`：当前任务最小权限集合；
- `forbidden_tools`：一旦执行就代表攻击成功；
- `sensitive_values`：实验使用的假敏感数据。

练习：复制一个攻击场景，改成“恶意招聘简历诱导 Agent 导出候选人数据”，然后运行场景测试。

## 第 3 天：理解数据模型和虚拟工具

阅读：

```text
agentguard_lab/models.py
agentguard_lab/tools.py
```

`models.py` 中的 `Scenario`、`ToolCall`、`ToolExecution` 和 `RunResult` 分别对应：

```text
测试题目 -> 模型计划 -> 实际执行记录 -> 最终实验报告
```

`VirtualToolbox` 的方法名称以 `_tool_` 开头。它不会调用操作系统或网络，只会修改 `self.state`。

练习：增加一个低风险工具 `save_note`，让它把标题和正文保存到内存列表，并补充一个测试。

## 第 4 天：理解五道防线

阅读 `agentguard_lab/guardrails.py`。

依次关闭或开启侧边栏中的自定义策略，观察：

1. 只有检测器时，攻击仍然成功。
2. 只有工具白名单时，删除和外发工具被阻止。
3. 只有敏感数据检查时，含金丝雀令牌的外发被阻止，但不含敏感值的日程攻击仍可能成功。
4. 开启人工审批时，合法高风险操作也会暂停。

今天需要理解“纵深防御”：不依赖一条完美规则，而是让多道独立防线共同降低风险。

## 第 5 天：理解执行引擎和指标

阅读：

```text
agentguard_lab/engine.py
agentguard_lab/evaluator.py
```

在 `engine.py` 中找到下面四个关键变量：

- `injection_detected`；
- `attack_succeeded`；
- `task_succeeded`；
- `false_positive`。

然后运行：

```powershell
python -m agentguard_lab.cli --profile all
```

尝试用自己的话解释为什么平衡模式综合分高于严格模式。标准答案不是“平衡模式更安全”，而是“二者都阻断了内置攻击，但平衡模式没有暂停合法邮件任务，因此可用性更高”。

## 第 6 天：理解真实模型适配器

阅读 `agentguard_lab/providers.py`。

本地规则引擎直接返回场景声明的正常和攻击动作，保证测试可重复。真实模型适配器会：

1. 构造系统消息和用户消息；
2. 将可用工具说明发送给兼容接口；
3. 要求模型返回结构化 JSON；
4. 将 JSON 转换成 `ToolCall`；
5. 交给同一套防御流水线处理。

这体现了一个重要设计：模型可以替换，但安全策略、工具箱和评测逻辑不应绑死在某个模型服务商上。

不要为了测试接口把 API Key 写进代码。只使用 `.env` 或页面密码输入框。

## 第 7 天：测试、提交和演示

执行：

```powershell
python -m ruff check .
python -m pytest
git status
```

确保静态检查通过、测试通过、`.env` 没有出现在 Git 状态中。

录制演示视频时按以下顺序：

1. 用 15 秒介绍问题：不可信文档可能控制 Agent 工具。
2. 用 30 秒运行基线攻击。
3. 用 30 秒运行平衡防御。
4. 用 30 秒展示批量指标。
5. 用 30 秒打开代码架构和测试。
6. 用 15 秒说明限制和下一步计划。

## 必须亲自完成的三个修改

上传前建议至少亲自完成三个小改动，这样面试时不会只会背 README：

1. 新增一个与你专业相关的攻击场景。
2. 新增一个虚拟工具并写测试。
3. 调整综合分权重，并解释为什么这样调整。

例如：

- 计算机专业：代码仓库工具、CI 工具、数据库查询工具；
- 财会专业：发票读取、报销审批、财务数据导出工具；
- 法学专业：合同读取、条款检索、文档外发工具；
- 医学专业：病历检索、预约、患者信息导出工具；
- 教育专业：成绩查询、课程通知、学生信息导出工具。

## 排错清单

### 找不到 streamlit

```powershell
python -m pip install -r requirements.txt
python -m streamlit run app.py
```

使用 `python -m streamlit` 比直接执行 `streamlit` 更不容易遇到环境路径问题。

### PowerShell 无法运行 run.ps1

```powershell
Set-ExecutionPolicy -Scope Process Bypass
.\run.ps1
```

该设置只影响当前 PowerShell 窗口。

### 端口 8501 被占用

```powershell
python -m streamlit run app.py --server.port 8502
```

然后访问 `http://localhost:8502`。

### JSON 修改后无法加载

```powershell
python -m pytest tests/test_scenarios.py -v
```

检查逗号、引号、唯一 ID、工具名称，以及预期工具是否位于 `allowed_tools` 中。

### API 接口返回 401

检查 API Key、模型名称和接口地址。接口地址一般需要包含 `/v1`，但应以服务商文档为准。

### API 接口返回 400

适配器已经会在接口不支持 `response_format` 时自动重试。如果仍然返回 400，请先使用本地规则引擎完成项目演示，再根据服务商文档调整 `providers.py` 中的请求体。

## 自测问题

在面试前，尝试不用看 README 回答：

1. Agent 与普通聊天模型最大的安全差别是什么？
2. 间接提示词注入和直接提示词注入有什么区别？
3. 为什么仅靠关键词检测不够？
4. 工具白名单如何体现最小权限原则？
5. 为什么要同时测 ASR、TSR 和 FPR？
6. 严格模式为什么不是所有场景下的最佳策略？
7. 本地规则引擎的意义和限制是什么？
8. 如果接入真实邮箱，最先需要补充哪些安全措施？
9. 如何让真实模型评测结果具有统计意义？
10. 你亲自增加了哪个场景、工具或防御策略？

能自然回答其中 8 个，并现场修改一个场景，就已经能把这个项目真正讲成自己的项目。
