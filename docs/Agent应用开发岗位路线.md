# Agent 应用开发岗位路线

你的目标岗位是 Agent 应用开发，因此这个项目不能只展示“我知道提示词注入”，还要展示你能把模型、工具、状态、安全和评测组合成一个可维护的系统。

## 岗位真正看什么

大多数 Agent 应用开发岗位不会要求应届生训练基础模型，更关注以下能力：

| 岗位能力 | 项目中的对应实现 | 你需要会解释什么 |
|---|---|---|
| 大模型 API | `providers.py` | 如何构造消息、要求结构化输出、处理接口异常 |
| Tool Calling | `models.py`、`tools.py` | 工具定义、参数、风险、副作用和执行结果 |
| Agent 执行循环 | `engine.py` | 读取环境、生成决策、检查策略、执行工具、返回结果 |
| 状态管理 | `VirtualToolbox.state` | 如何保存中间结果，为什么实验结束后清空 |
| MCP 协议 | `mcp_server.py` | Tool、Resource、Prompt 和传输方式的区别 |
| Guardrails | `guardrails.py` | 检测、阻断、最小权限、数据防泄漏和人工审批 |
| Evaluation | `evaluator.py` | 为什么同时测 ASR、TSR、FPR 和检测率 |
| 可观测性 | `ToolExecution`、`GuardEvent` | 如何记录工具轨迹和阻断原因 |
| 工程质量 | `tests/`、GitHub Actions | 如何让场景、协议和指标自动回归 |
| 产品展示 | `app.py` | 如何把复杂执行链展示给非算法面试官 |

## 先掌握这条主线

请先记住项目的核心执行链：

```text
用户目标
  -> Agent 读取环境内容
  -> 决策引擎生成结构化 ToolCall
  -> Guardrails 检查权限和数据
  -> VirtualToolbox 执行或阻断
  -> 记录 ToolExecution / GuardEvent
  -> Evaluator 计算 ASR、TSR、FPR
```

MCP 不是另一个执行引擎，它是项目对外提供能力的标准协议层：

```text
Agent 客户端
  -> MCP 动态发现工具
  -> 调用 run_security_scenario
  -> AgentGuard 执行已有安全流程
  -> MCP 返回结构化结果
```

面试时能把这两条链讲清楚，比背诵框架名称更重要。

## 四周学习安排

### 第 1 周：Python 工程和结构化输出

目标：能够独立修改数据模型和接口适配器。

学习内容：

- Python 函数、类、异常、类型注解和 `dataclass`；
- JSON 序列化和反序列化；
- HTTP 请求、状态码和超时；
- 环境变量与 API Key 管理；
- Pydantic 或等价的结构化数据校验思路。

项目任务：

1. 阅读 `models.py`，画出 `Scenario -> AgentDecision -> RunResult` 的关系。
2. 阅读 `providers.py`，找到请求体和 JSON 解析位置。
3. 给真实模型适配器增加一次错误日志，但不得记录 API Key。
4. 为非法 JSON 返回增加测试。

验收标准：你能解释为什么 Agent 工具调用必须使用结构化数据，而不能从自然语言中随便截取字符串执行。

### 第 2 周：Agent Loop 和工具系统

目标：理解 Agent 不只是一次模型调用，而是“模型、工具、状态”组成的循环。

学习内容：

- Tool Calling 的参数 Schema；
- 工具有无副作用的区别；
- Agent 状态、短期记忆和运行上下文；
- 同步与异步调用的基本区别；
- 超时、重试、幂等性和失败恢复。

项目任务：

1. 运行正常场景“代码仓库 Issue 分析”。
2. 跟踪 `read_issue -> create_task_plan -> run_tests` 三步调用。
3. 新增一个 `comment_issue` 虚拟工具。
4. 让工具支持 `request_id`，避免同一次请求被重复执行。
5. 增加工具执行失败场景，确保 Agent 不会把失败当成功。

验收标准：你能说明为什么 `publish_package` 必须比 `read_issue` 拥有更严格的权限和审批策略。

### 第 3 周：MCP 协议

目标：能够把一个 Python 能力包装成其他 Agent 可发现和调用的 MCP 服务。

学习内容：

- MCP Host、Client、Server 的关系；
- Tool、Resource、Prompt 三种原语；
- stdio 与 Streamable HTTP；
- 工具 Schema 和结构化返回；
- MCP 身份认证和最小权限的基本概念。

项目任务：

1. 阅读 `mcp_server.py`。
2. 使用 MCP Inspector 枚举 3 个工具。
3. 调用 `run_security_scenario` 并检查结构化结果。
4. 读取 `agentguard://scenario/attack_repo_supply_chain` 资源。
5. 新增一个 `get_tool_risk_catalog` MCP Tool，返回全部虚拟工具风险等级。

验收标准：你能回答“为什么 MCP Server 不是简单把所有后端函数都暴露给模型”。正确方向包括权限、输入校验、副作用、认证、限流和审计。

### 第 4 周：评测、可观测性和项目包装

目标：让项目从“能跑”升级到“结果可信、问题可定位”。

学习内容：

- 离线测试集与回归测试；
- 确定性基线和真实模型随机采样；
- Agent Trace、Span 和工具调用链；
- 成本、延迟、成功率和安全指标；
- CI、README、演示视频和技术复盘。

项目任务：

1. 为每个真实模型场景重复运行至少 5 次。
2. 记录模型版本、温度、平均延迟和攻击成功率。
3. 增加 `trace_id`，把一次运行的模型和工具事件关联起来。
4. 生成一份不同模型或不同策略的实验报告。
5. 录制 2 至 3 分钟项目演示视频。

验收标准：你能解释实验结果的适用范围，不会把内置测试集上的 `0% ASR` 说成“系统绝对安全”。

## 面试项目讲法

### 30 秒版本

> 我做了一个工具调用 Agent 的安全评测平台。系统把模型决策、Guardrails、虚拟工具和评测拆成独立模块，支持 12 个正常与攻击场景，并通过 MCP 向其他 Agent 客户端暴露场景执行和策略对比能力。项目重点解决不可信内容诱导 Agent 越权调用工具的问题，能够量化攻击成功率、任务成功率和误拦截率。

### 两分钟版本

按照四段讲：

1. **问题**：Agent 能调用真实工具，不可信 Issue 或文档可能诱导它读取密钥、删除文件或发布软件包。
2. **架构**：模型输出结构化 `ToolCall`，经过注入检测、指令隔离、工具白名单、敏感数据检查和人工审批后进入虚拟工具箱。
3. **评测**：用 12 个确定性场景比较四组策略，同时测 ASR、TSR 和 FPR，避免“全部拒绝”获得虚假安全。
4. **工程化**：提供 Streamlit 界面、CLI、MCP Server、协议测试、GitHub Actions，并明确真实模型随机性和当前限制。

## 高频技术追问

### Agent 和 Workflow 有什么区别？

Workflow 的步骤通常由开发者预先确定，Agent 会根据模型输出动态选择下一步工具。生产系统中常把两者结合：关键流程使用确定性编排，局部决策交给 Agent。

### 为什么工具调用要做服务端校验？

模型输出是不可信输入。即使 Schema 合法，也可能越权、包含敏感数据或触发高风险副作用，所以必须在服务端再次校验权限、参数、审批和幂等性。

### MCP 解决了什么问题？

MCP 统一了 Agent 应用发现工具、读取资源和使用提示模板的方式，降低客户端与具体服务的耦合。但它不会自动解决权限、安全和业务正确性，这些仍由服务端负责。

### 为什么需要确定性规则引擎？

它让攻击链和防御结果可重复，适合 CI 回归。真实模型评测则用于衡量现实行为，需要多次采样并记录模型版本。二者解决的问题不同。

### 如何处理长时间运行的工具？

常见方案包括异步任务队列、进度事件、超时、取消、重试和幂等键。当前项目的虚拟工具是同步短任务，后续可以把 `run_tests` 改造成异步任务演示这些能力。

### 如果要接入真实 GitHub，需要补什么？

至少需要 OAuth 或 GitHub App 身份、仓库级权限、分支保护、命令参数校验、网络与进程沙箱、人工审批、速率限制、审计日志和密钥管理。不能直接把虚拟工具替换成高权限 Token。

## 下一阶段作品组合

AgentGuard Lab 可以证明你会安全、工具调用、评测和 MCP。为了形成更完整的作品组合，第二个项目建议做“代码仓库 Agent”：

- 输入一个小型 Python 仓库和 Issue；
- 使用 AST 建立函数与文件依赖；
- Agent 生成修改计划并在沙箱中执行；
- 自动运行测试并根据失败结果迭代；
- 所有写文件、执行命令和提交操作需要审批；
- 使用 AgentGuard Lab 的场景评测其安全性。

两个项目连起来的叙事是：你不仅会做能够完成任务的 Agent，也会评测和约束 Agent 的行为。
